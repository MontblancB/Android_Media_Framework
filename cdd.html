📜 Google CDD

Android Compatibility Definition Document

1 Overview: CDD란?

CDD (Compatibility Definition Document)는 Android 호환 디바이스가 되기 위해 반드시 충족해야 하는 요구사항을 정의한 정책 문서입니다.

🎯 목적

호환성 정책 정의

  • "Android 호환"의 의미 정의
  • 모호성 제거 및 명확한 기준 제시
  • 일관된 앱 생태계 보장
  • 하드웨어/소프트웨어 최소 사양 규정

⚖️ Policy vs Mechanism

CDD와 CTS의 역할

  • CDD (Policy): 무엇을 해야 하는가? (규정)
  • CTS (Mechanism): 규정을 지켰는가? (검증)
  • CDD는 법전, CTS는 시험

🌍 생태계 참여

GMS 인증의 기초

  • Google Play Store 탑재 조건
  • Android 상표 사용 권한
  • 개발자에게 안정적 타겟 제공
  • 사용자 경험 파편화 방지

📅 버전별 관리

Android 버전 대응

  • 각 Android 버전마다 별도 CDD 존재
  • Android 12 CDD, Android 13 CDD 등
  • 새로운 기능 및 API 반영
  • 하드웨어 발전 수용

3 요구사항 언어 (RFC2119)

CDD는 요구사항의 강제성을 명확히 하기 위해 RFC2119 표준 키워드를 사용합니다.

MUST / REQUIRED

필수 사항

  • 반드시 준수해야 함
  • 위반 시 호환되지 않음
  • CTS 테스트 실패 요인
  • 예: "Device implementations MUST support H.264 decoding."

SHOULD / RECOMMENDED

권장 사항

  • 강제는 아니지만 강력히 권장됨
  • 타당한 이유가 없다면 준수해야 함
  • 향후 버전에서 MUST로 변경될 수 있음
  • 예: "Device implementations SHOULD support 5GHz Wi-Fi."

MAY / OPTIONAL

선택 사항

  • 제조사의 선택에 따름
  • 준수하지 않아도 호환성에 문제 없음
  • 기능 차별화 요소
  • 예: "Device implementations MAY include a hardware keyboard."
🆔 Requirement ID:

모든 "MUST" 요구사항은 고유한 ID를 가집니다. (예: [C-1-1])
이를 통해 특정 요구사항을 정확히 참조하고 추적할 수 있습니다.

미디어 프레임워크 관점에서 가장 중요한 멀티미디어 섹션의 요구사항입니다.

🎬 필수 코덱

MUST Support

  • Video: H.264 (AVC), H.265 (HEVC), VP8, VP9
  • Audio: AAC LC, HE-AAC, FLAC, MP3, Opus
  • MediaCodecList를 통해 선언 필수

🚀 Performance Class

고성능 미디어 기준 (Android 12+)

  • Class 11 (R), Class 12 (S) 등
  • 4K@30fps, 1080p@60fps 녹화
  • 동시 코덱 세션 수 보장
  • 앱이 런타임에 쿼리 가능

📺 디바이스별 차이

폼팩터 특화

  • Android TV: H.265/VP9 4K 디코딩 필수
  • Handheld: 카메라 녹화 성능 중시
  • Automotive: 오디오 라우팅 및 멀티존

5 CDD vs CTS

CDD(정책)와 CTS(검증)의 관계를 명확히 이해하는 것이 중요합니다.

sequenceDiagram participant GOOGLE as Google participant OEM as OEM/Manufacturer participant DEVICE as Device Note over GOOGLE: 1. CDD 발행 (Policy) GOOGLE->>OEM: Android 14 CDD 문서 제공 Note over OEM: 2. 구현 (Implementation) OEM->>DEVICE: CDD 요구사항에 맞춰
HW/SW 개발 Note over GOOGLE: 3. CTS 배포 (Mechanism) GOOGLE->>OEM: CTS 테스트 스위트 제공 Note over OEM: 4. 검증 (Verification) OEM->>DEVICE: CTS 실행 DEVICE-->>OEM: Pass/Fail 결과 Note over OEM: 5. 인증 (Certification) OEM->>GOOGLE: CTS 리포트 제출 GOOGLE-->>OEM: GMS 인증 승인

📜 CDD (Document)

  • 형태: PDF/HTML 문서
  • 내용: 요구사항 명세
  • 주체: 사람이 읽고 해석
  • 역할: 기준점 제시

✅ CTS (Software)

  • 형태: 테스트 코드 (APK)
  • 내용: 자동화된 테스트 케이스
  • 주체: 컴퓨터가 실행
  • 역할: 기준 충족 여부 판별

6 실제 CDD 요구사항 예제

CDD 문서의 실제 Requirement ID와 구체적인 예시를 설명합니다.

📋 미디어 관련 MUST 요구사항

Requirement ID 요구사항 CTS 검증
[C-1-1] Handheld device implementations MUST support H.264 decoding at 1080p@30fps MediaCodecTest#testH264Decoder1080p30fps
[C-1-2] Device implementations MUST support AAC LC decoding at 48kHz, stereo MediaCodecTest#testAacLcDecoder
[C-5-1] Television implementations MUST support H.265 decoding at 4K@30fps MediaCodecTest#testHevcDecoder4K30fps
[C-5-3] Television implementations MUST support VP9 Profile 2 (10-bit) decoding MediaCodecTest#testVp9Profile2Decoder
[C-3-2] Automotive implementations MUST support audio output latency ≤ 100ms AudioLatencyTest#testOutputLatency
[C-8-1] Watch implementations MUST support Opus audio codec MediaCodecTest#testOpusDecoder

📝 Requirement ID 형식

🔢 ID 구조

[C-X-Y] 형식

  • C: Conditional (조건부 요구사항)
  • X: Device Type (1=Handheld, 3=Automotive, 5=TV, 8=Watch)
  • Y: 일련번호 (섹션 내 순서)

📌 예시

  • [C-1-3]: Handheld 디바이스의 3번째 조건부 요구사항
  • [C-5-7]: Television 디바이스의 7번째 조건부 요구사항
  • [C-0-1]: 모든 디바이스 타입에 적용되는 요구사항

⚠️ 위반 시 결과

  • MUST 위반: CTS 테스트 실패
  • SHOULD 위반: 경고만 발생, 통과 가능
  • MAY 위반: 아무 영향 없음
  • GMS 인증: MUST 모두 통과 필수

🔍 추적 가능성

  • CDD 문서에서 Requirement ID로 검색
  • CTS 소스코드에 ID가 주석으로 포함
  • 실패 시 어떤 요구사항 위반인지 명확
  • 버전별 변경 이력 추적 용이
💡 실제 CDD 문서 예시: 
5.1.3. Video Codecs

If device implementations support H.264 decoders, they:
* [C-1-1] MUST support Baseline Profile Level 3.
* [C-1-2] MUST support the SD (Standard Definition) video decoding profiles
  in the following table and be able to decode videos with sizes up to the
  maximum supported resolution.
* [C-1-3] SHOULD support Main Profile Level 4.
* [C-1-4] SHOULD support the HD (High Definition) video decoding profiles.

[C-0-3] MUST

  • Runtime Permissions 구현
  • 사용자 명시적 동의 필수
  • 권한 철회 가능
  • Dangerous Permissions 관리

✅ Verified Boot

[C-0-4] MUST

  • Bootloader 검증
  • System 파티션 검증
  • dm-verity 활성화
  • 변조 감지 및 경고

🔑 생체인식 보안

보안 등급 FAR (False Accept Rate) 용도 예시
Class 3 (Strong) < 1 / 50,000 결제, 앱 잠금 해제 초음파 지문, 3D 얼굴인식
Class 2 (Weak) < 1 / 1,000 디바이스 잠금 해제 광학 지문, 2D 얼굴인식
Class 1 (Convenience) < 1 / 100 빠른 잠금 해제 스마트 워치 얼굴인식

🛡️ DRM & 미디어 보안

🔒 Widevine L1

하드웨어 보안 필수

  • TEE (Trusted Execution Environment)
  • 하드웨어 백업 키스토어
  • Secure Video Path (SVP)
  • HDCP 2.2+ 지원

🔓 Widevine L3

소프트웨어 보안

  • 최대 SD (480p) 해상도
  • TEE 불필요
  • 모든 디바이스 지원 가능
  • 저비용 솔루션

🎬 PlayReady

Microsoft DRM

  • SL2000: 하드웨어 보안 (L1 상당)
  • SL3000: 소프트웨어 보안 (L3 상당)
  • Xbox, Windows 플랫폼
  • DASH/HLS 지원

🍎 FairPlay

Apple DRM (참고용)

  • iOS/macOS/tvOS 전용
  • 하드웨어 보안 통합
  • HLS 스트리밍
  • Android 미지원
⚠️ 보안 CTS 테스트:
  • CtsSecurityTestCases: SELinux, 권한 모델, 암호화 검증
  • CtsVerifiedBootTestCases: Verified Boot, dm-verity 검증
  • CtsBiometricsTestCases: 생체인식 FAR/FRR 테스트

⚖️ MUST vs SHOULD 의사결정 가이드

✅ MUST 구현 우선순위

  • 1순위: 플랫폼 기본 기능 (H.264, AAC)
  • 2순위: 디바이스 타입 필수 (TV HEVC)
  • 3순위: 버전별 신규 필수 (Android 14 AV1)
  • 검증: CTS 실패 시 GMS 인증 불가

⚠️ SHOULD 구현 전략

  • 비용 고려: 하드웨어 제약, 개발 공수
  • 시장 요구: 타겟 사용자 분석
  • 차별화: 경쟁사 대비 강점
  • 미래 대비: 다음 버전 MUST 예상

🚀 권장 구현 SHOULD

  • HEVC 디코딩 (콘텐츠 호환성)
  • Widevine L1 (프리미엄 스트리밍)
  • HDR10 (프리미엄 디스플레이)
  • 60fps 인코딩 (고품질 녹화)

💰 비용 대비 효과

  • 높음: HE-AAC (소프트웨어)
  • 중간: HEVC (라이선스 필요)
  • 낮음: 8K 디코딩 (제한적 사용)
  • 평가: ROI 분석 필수
⚠️ 주의사항:
  • SHOULD는 권장이지만, 구현하지 않으면 사용자 경험 저하 가능
  • 경쟁사가 SHOULD 구현 시 시장에서 불리할 수 있음
  • 다음 Android 버전에서 SHOULD → MUST로 변경될 수 있음 (AV1 사례)
  • Performance Class 선언 시 모든 관련 SHOULD가 MUST로 전환

15 디버깅 & 트러블슈팅

CDD 요구사항 위반 시 체계적인 디버깅 방법을 설명합니다.

🔍 CTS 실패 → CDD 매핑 

# CTS 실패 로그 예시
FAIL: testH264Decoder4K30fps
AssertionError: [C-5-1] TV implementations MUST
support H.264 decoding at 3840x2160@30fps

# CDD 문서 검색
1. Android TV CDD 문서 열기
2. Ctrl+F → "[C-5-1]" 검색
3. Section 5.1.3 Video Codecs 확인
4. 구체적 요구사항 파악

📋 체크리스트

  • ✓ 디바이스 타입 확인 (Handheld/TV/Auto/Watch)
  • ✓ Android 버전 확인 (버전별 요구사항 다름)
  • ✓ Performance Class 선언 여부
  • ✓ User Build인지 확인 (eng/userdebug 금지)

🛠️ 설정 파일 검증 

# media_codecs.xml 확인
adb pull /vendor/etc/media_codecs.xml

# 코덱 capabilities 확인
<Limit name="size" min="96x96" max="3840x2160" />
<Limit name="frame-rate" range="1-60" />

# media_profiles.xml 검증
adb pull /vendor/etc/media_profiles.xml

💻 런타임 검증 

# MediaCodecList 덤프
adb shell dumpsys media.codec

# Performance Class 확인
adb shell getprop ro.odm.build.media_performance_class

# 시스템 속성 확인
adb shell getprop | grep media

🔧 일반적인 문제 및 해결법

문제 원인 해결법
H.264 디코더 누락 media_codecs.xml에 미등록 <MediaCodec name="OMX.google.h264.decoder" type="video/avc" /> 추가
해상도 제한 위반 max resolution 설정 부족 <Limit name="size" max="3840x2160" />로 수정
Performance Class 불일치 하드웨어 성능 부족 Performance Class 선언 제거 또는 하드웨어 업그레이드
DRM 테스트 실패 Widevine provisioning 안됨 adb shell pm list packages | grep drm로 CDM 확인
SELinux 위반 Permissive 모드 또는 정책 누락 adb shell getenforce로 Enforcing 확인 및 정책 수정
✅ 디버깅 프로세스:
  1. CTS 실패 로그에서 Requirement ID 추출 ([C-X-Y])
  2. 해당 버전의 CDD PDF에서 Requirement ID 검색
  3. 정확한 요구사항 확인 (해상도, 프레임레이트 등)
  4. media_codecs.xml 및 시스템 속성 확인
  5. 설정 수정 또는 HAL 구현 업데이트
  6. CTS 재실행 및 검증
