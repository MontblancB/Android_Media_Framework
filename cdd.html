Android Compatibility Definition Document

1 Overview: CDD란?

CDD (Compatibility Definition Document)는 Android 호환 디바이스가 되기 위해 반드시 충족해야 하는 요구사항을 정의한 정책 문서입니다.

🎯 목적

호환성 정책 정의

  • "Android 호환"의 의미 정의
  • 모호성 제거 및 명확한 기준 제시
  • 일관된 앱 생태계 보장
  • 하드웨어/소프트웨어 최소 사양 규정

⚖️ Policy vs Mechanism

CDD와 CTS의 역할

  • CDD (Policy): 무엇을 해야 하는가? (규정)
  • CTS (Mechanism): 규정을 지켰는가? (검증)
  • CDD는 법전, CTS는 시험

🌍 생태계 참여

GMS 인증의 기초

  • Google Play Store 탑재 조건
  • Android 상표 사용 권한
  • 개발자에게 안정적 타겟 제공
  • 사용자 경험 파편화 방지

📅 버전별 관리

Android 버전 대응

  • 각 Android 버전마다 별도 CDD 존재
  • Android 12 CDD, Android 13 CDD 등
  • 새로운 기능 및 API 반영
  • 하드웨어 발전 수용

CDD는 디바이스의 다양한 측면을 포괄하는 섹션들로 구성되어 있습니다.

💡 주요 섹션 설명:
  • Device Types: 폼팩터별(폰, TV, 차, 시계) 특화 요구사항
  • Multimedia: 코덱 지원, 디코딩/인코딩 성능, 오디오 레이턴시 등
  • Hardware: 디스플레이 해상도, 카메라 사양, 센서 정확도 등
  • Security: 권한 모델, 샌드박스, 암호화, 생체인식 보안 등

3 요구사항 언어 (RFC2119)

CDD는 요구사항의 강제성을 명확히 하기 위해 RFC2119 표준 키워드를 사용합니다.

MUST / REQUIRED

필수 사항

  • 반드시 준수해야 함
  • 위반 시 호환되지 않음
  • CTS 테스트 실패 요인
  • 예: "Device implementations MUST support H.264 decoding."

SHOULD / RECOMMENDED

권장 사항

  • 강제는 아니지만 강력히 권장됨
  • 타당한 이유가 없다면 준수해야 함
  • 향후 버전에서 MUST로 변경될 수 있음
  • 예: "Device implementations SHOULD support 5GHz Wi-Fi."

MAY / OPTIONAL

선택 사항

  • 제조사의 선택에 따름
  • 준수하지 않아도 호환성에 문제 없음
  • 기능 차별화 요소
  • 예: "Device implementations MAY include a hardware keyboard."
🆔 Requirement ID:

모든 "MUST" 요구사항은 고유한 ID를 가집니다. (예: [C-1-1])
이를 통해 특정 요구사항을 정확히 참조하고 추적할 수 있습니다.

4 멀티미디어 호환성

미디어 프레임워크 관점에서 가장 중요한 멀티미디어 섹션의 요구사항입니다.

🎬 필수 코덱

MUST Support

  • Video: H.264 (AVC), H.265 (HEVC), VP8, VP9
  • Audio: AAC LC, HE-AAC, FLAC, MP3, Opus
  • MediaCodecList를 통해 선언 필수

🚀 Performance Class

고성능 미디어 기준 (Android 12+)

  • Class 11 (R), Class 12 (S) 등
  • 4K@30fps, 1080p@60fps 녹화
  • 동시 코덱 세션 수 보장
  • 앱이 런타임에 쿼리 가능

📺 디바이스별 차이

폼팩터 특화

  • Android TV: H.265/VP9 4K 디코딩 필수
  • Handheld: 카메라 녹화 성능 중시
  • Automotive: 오디오 라우팅 및 멀티존

5 CDD vs CTS

CDD(정책)와 CTS(검증)의 관계를 명확히 이해하는 것이 중요합니다.

sequenceDiagram participant GOOGLE as Google participant OEM as OEM/Manufacturer participant DEVICE as Device Note over GOOGLE: 1. CDD 발행 (Policy) GOOGLE->>OEM: Android 14 CDD 문서 제공 Note over OEM: 2. 구현 (Implementation) OEM->>DEVICE: CDD 요구사항에 맞춰
HW/SW 개발 Note over GOOGLE: 3. CTS 배포 (Mechanism) GOOGLE->>OEM: CTS 테스트 스위트 제공 Note over OEM: 4. 검증 (Verification) OEM->>DEVICE: CTS 실행 DEVICE-->>OEM: Pass/Fail 결과 Note over OEM: 5. 인증 (Certification) OEM->>GOOGLE: CTS 리포트 제출 GOOGLE-->>OEM: GMS 인증 승인

📜 CDD (Document)

  • 형태: PDF/HTML 문서
  • 내용: 요구사항 명세
  • 주체: 사람이 읽고 해석
  • 역할: 기준점 제시

✅ CTS (Software)

  • 형태: 테스트 코드 (APK)
  • 내용: 자동화된 테스트 케이스
  • 주체: 컴퓨터가 실행
  • 역할: 기준 충족 여부 판별
