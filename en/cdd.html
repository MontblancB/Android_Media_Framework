1 Overview: What is CDD?

CDD (Compatibility Definition Document) is a policy document that defines the requirements that must be met for a device to be Android compatible.

üéØ Purpose

Compatibility Policy Definition

  • Defines what "Android compatible" means
  • Eliminates ambiguity and provides clear standards
  • Ensures a consistent app ecosystem
  • Specifies minimum hardware/software requirements

‚öñÔ∏è Policy vs Mechanism

Roles of CDD and CTS

  • CDD (Policy): What must be done? (Specification)
  • CTS (Mechanism): Was the specification followed? (Verification)
  • CDD is the rulebook, CTS is the exam

üåç Ecosystem Participation

Foundation of GMS Certification

  • Requirement for Google Play Store inclusion
  • Android trademark usage rights
  • Provides stable targets for developers
  • Prevents user experience fragmentation

üìÖ Version Management

Android Version Mapping

  • Separate CDD exists for each Android version
  • Android 12 CDD, Android 13 CDD, etc.
  • Reflects new features and APIs
  • Accommodates hardware advancements

3 Requirement Language (RFC2119)

CDD uses RFC2119 standard keywords to clearly indicate the enforceability of requirements.

MUST / REQUIRED

Mandatory

  • Must be complied with
  • Non-compliance means incompatible
  • Causes CTS test failure
  • Example: "Device implementations MUST support H.264 decoding."

SHOULD / RECOMMENDED

Recommended

  • Not mandatory but strongly recommended
  • Should comply unless there's a valid reason
  • May become MUST in future versions
  • Example: "Device implementations SHOULD support 5GHz Wi-Fi."

MAY / OPTIONAL

Optional

  • Left to manufacturer's discretion
  • Non-compliance doesn't affect compatibility
  • Feature differentiation element
  • Example: "Device implementations MAY include a hardware keyboard."
üÜî Requirement ID:

All "MUST" requirements have a unique ID. (e.g., [C-1-1])
This allows specific requirements to be precisely referenced and tracked.

üî¢ ID Structure

[C-X-Y] format

  • C: Conditional (conditional requirement)
  • X: Device Type (1=Handheld, 3=Automotive, 5=TV, 8=Watch)

  • [C-0-1]: Requirement applicable to all device types

  • MUST violation: CTS test failure
  • SHOULD violation: Warning only, can still pass
  • MAY violation: No impact
  • GMS certification: All MUST requirements must pass

  • ID included as comments in CTS source code
  • Clear which requirement is violated on failure
  • Easy to track version-specific change history

‚öñÔ∏è MUST vs SHOULD Decision Guide

‚úÖ MUST Implementation Priority

  • Priority 1: Platform basic features (H.264, AAC)
  • Priority 2: Device type mandatory (TV HEVC)
  • Priority 3: Version-specific new mandatory (Android 14 AV1)
  • Verification: CTS failure means no GMS certification

‚ö†Ô∏è SHOULD Implementation Strategy

  • Cost consideration: Hardware constraints, development effort
  • Market demand: Target user analysis
  • Differentiation: Competitive advantage
  • Future-proofing: Anticipate next version MUST

üöÄ Recommended SHOULD Implementations

  • HEVC decoding (content compatibility)
  • Widevine L1 (premium streaming)
  • HDR10 (premium display)
  • 60fps encoding (high-quality recording)

üí∞ Cost-Effectiveness

  • High: HE-AAC (software)
  • Medium: HEVC (license required)
  • Low: 8K decoding (limited use)
  • Evaluation: ROI analysis required
‚ö†Ô∏è Considerations:
  • SHOULD is recommended, but not implementing may degrade user experience
  • May be disadvantaged in market if competitors implement SHOULD
  • SHOULD may change to MUST in next Android version (AV1 case)
  • When declaring Performance Class, all related SHOULD become MUST
